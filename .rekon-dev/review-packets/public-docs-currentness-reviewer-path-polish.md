## PURPOSE PRESERVATION CHECK

- Original problem: Rekon now has a public product entry point, but the docs tree
  still mixes current guidance with historical slice records and generated
  freshness output.
- Product guarantee: external reviewers can quickly find what Rekon does today,
  how to run it, what to inspect first, and where historical rigor lives without
  confusing it for the getting-started path.
- Boundary guarantee: the docs do not over-promise automation. Context, proof,
  approval, execution, and generated publications remain separate.
- What would mean we failed: reviewers land in strategy slice history before the
  README/demo path; generated `docs/INDEX.md` becomes the human landing page;
  docs imply Rekon writes source, approves plans, executes commands, or runs
  Circe in the demo.
- Regression check: `tests/docs/public-docs-currentness.test.mjs` asserts the
  public path, demo commands, status labels, historical separation, changelog,
  and review packet.

## REVIEWER PATH

If a reviewer has 10 minutes:

1. Read the root README and `docs/README.md`.
2. Run the 5-minute demo or the deeper task-context-to-handoff demo.
3. Inspect `TaskContextReport`, `ResolverPacket`,
   `IntentPlanActionabilityReport`, `WorkOrder`, `VerificationPlan`, and
   `Publication` artifacts.
4. Run `artifacts validate`.
5. Read the safety model and intent bundle handoff workflow guide.

## DOCS INDEX DECISION

- Decision: regenerate and commit `docs/INDEX.md` if the final freshness command
  changes it.
- Reason: `docs/INDEX.md` is tracked, deterministic, generated by
  `rekon docs freshness`, and linked from `docs/README.md` as a generated
  freshness index.
- Boundary: it remains generated status output, not the human landing page.

## BOUNDARY MODEL

- Context is not proof.
- Generated publications are not canonical truth.
- Plan review is not approval.
- Verification hints are not command execution.
- Rekon does not write source, approve plans, execute verification commands, or
  run Circe in the public demo.
