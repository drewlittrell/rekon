# Roadmap

Rekon is currently version `1.0.0`. The core public shape is in place: typed
artifacts, SDK capabilities, runtime, CLI, repository scan, model projection,
policy evaluation, resolver traces, publications, memory, intent artifacts,
verification recording, reconciliation planning, and artifact validation.

## Current Model Interface

- Production scan and refresh index JavaScript, TypeScript, and Python source.
  Resolved provider evidence feeds the shared capability graph, and a measured
  Python service task now receives its three constructor collaborators and
  focused test under the compact context budget.
- A shared, budget-aware compiler serves CLI and MCP task context.
- A shared task-operation policy classifies local, cross-file, cross-system,
  contract-changing, migration, and critical-flow work. It keeps complete work
  on the smallest sufficient context profile, routes high-risk work through the
  existing intent work-order flow, and reserves deep context for an unexplained
  validation failure.
- Compact delivery can carry one exact deterministic, source-digest-bound
  `sourceSpans` anchor per delivered path under profile-level count and
  character caps. Extension and placement tasks may also receive one
  inference-selected repository exemplar with a deterministic bound excerpt.
  These have source-backed contract coverage but no accepted model-efficiency
  claim yet.
- A minimal delivery adapter serves model-facing MCP, CLI, and evaluation
  payloads while the full packet remains available for audit.
- Coding agents see three source-safe MCP tools: task context, exact
  source-target resolution, and post-edit change validation. Task context may
  refresh `.rekon/` artifacts; source-target resolution and validation are
  read-only. Operator-facing orientation,
  placement, and preflight remain available through the CLI and compatibility
  calls.
- MCP and CLI resolve named dependency, dependent, test, contract, consumer,
  producer, and implementation targets. Resolution requires an exact
  identifier exposed by inspected source, excludes already-read paths, and
  does not silently broaden to text or semantic search.
- Managed repositories can keep a bounded Rekon bootstrap in `AGENTS.md`
  without replacing human-authored guidance.
- Whole-file exports cannot replace protected agent instruction files.
- A 17-task, source-free calibration gate measures required-context recall,
  allowlisted precision, irrelevant paths, pact/check retention, freshness
  warnings, and projected token cost without provider calls.
- Digest-bound maintainer judgments prevent a changed context selection from
  inheriting an earlier qualitative approval.
- A credential-free local-agent harness compares actual isolated changes with
  and without Rekon context. New runs retain source-free aggregate Codex usage
  and a separate visible-payload estimate before any API canary is needed.
- A managed-interface gate verifies that agents request task context through
  MCP or CLI before broad exploration instead of relying on prompt injection.
- Task context selects path-matched `CapabilityContract` rules as declared,
  freshness-aware pacts and checks instead of relying on verbose task prompts.
- Committed system and flow law compiles into an effective registry and a
  task-specific `TaskPact`; CLI and MCP share the same selection, including
  post-compaction rehydration and work-order lineage.
- Post-edit validation compares declared task paths with Git/current source,
  applies TaskPact scope, ownership, capability dependency policy, and complete
  flow handoffs, and returns typed proof obligations without executing checks.
  Selected VerificationResults and runtime observations bind to the exact
  checks and handoff edges they support. The acting model handles only declared
  semantic judgments; repository purpose and user outcomes are completion
  gates rather than advisory text.
- Targeted verification now preserves declared task and contract checks, fills
  changed-source test gaps only from passed isolated coverage linked to exact
  runner commands, and chooses a deterministic minimal candidate set. Failed,
  stale, incomplete, and missing checks return proof-local corrective context
  with bounded redacted diagnostics. Historical observations select commands;
  current digest-bound runs provide proof.
- A satisfied `ProofGateReport` binds those results to post-edit source digests.
  Executed verification captures the plan's bounded source before and after
  commands, refuses proof when those bytes changed, and carries the stable
  digest through `VerificationResult` into the final gate. `validate_change`
  compares exact digests rather than timestamps. Managed agents use the report
  for one incremental refresh; later source changes make the gate unusable.
  Existing adopted contracts reconcile against the new projection before the
  current snapshot, while first scans create no implicit repository law.
- Proof-gated refresh now completes accepted-change maintenance in one command.
  The gate roots observation lineage; absent baselines promote to a full first
  observation; stale embedding cache records are excluded; every maintained
  local publication and the Rekon-managed instruction block regenerate; and a
  final source-state recheck prevents mid-refresh edits from advancing the
  repository generation. Confirmed contract drift blocks acceptance.
- Contract maintenance is resumable by the coding agent: deterministic
  discovery returns a source-cited judgment task, and the same command can
  bind that judgment, perform permissioned adoption, compile, and reconcile.
- A source-free repository-law gate covers compositional system semantics,
  cross-system baton preservation, and unrelated-law isolation. It raises
  required-path recall from 72.22% to 100% at 100% precision, with a 282-token
  maximum compact delivery. A three-repeat Sol/xhigh managed evaluation passed
  6/6 Rekon changes versus 4/6 baselines and reduced average explored paths
  from 17.0 to 7.17. Reported tokens rose 36.3% and the visible estimate rose
  12.2%, so the accepted result is correctness and exploration gain with token
  overhead.
- An opt-in tiered-delivery probe preserved quality in four Sol/xhigh Rekon
  runs but reduced no actual source reads: all 16 conditional routes were still
  inspected. The full policy remains the default. The result is retained as
  historical after the managed instruction interface changed.
- Task-context routes now carry deterministic roles, necessity classes, and
  concrete reasons. The opt-in `role-aware` policy retained all 30 selected
  routes in its six-case source-free gate while reducing the initial mandatory
  set from 25 to 15. No model ran, so the result is route separation only; the
  full policy remains the production default.
- An independent configuration-rollout canary ran three Sol/xhigh repeats per
  policy. Both policies passed every scope, check, oracle, and MCP-adoption
  gate, but Sol read both conditional routes in every role-aware run.
  Exploration was unchanged; reported tokens rose 17.8% and the visible
  estimate rose 7.3%. Those routes are required by this task's oracle, so the
  result validates conditional access but not optional-route skipping. The
  retained model record is historical after the managed interface changed.
- A follow-up additive-helper canary made two selected routes explicitly
  optional in the task oracle. Both policies passed 3/3 Sol/xhigh runs, but
  role-aware used all six optional-route opportunities while full left one
  caller unused. Exploration was unchanged; role-aware used 14.1% more reported
  tokens, 3.2% more visible-estimate tokens, and 6.7% more commands. Conditional
  labels alone are therefore not a context-reduction mechanism. The retained
  model record is historical after the managed interface changed.
- A pathless-summary follow-up removed both optional paths from delivery and
  replaced them with resolved relationship summaries. Two three-run Sol/xhigh
  variants again passed every quality and adoption gate, but both inspected all
  six optional-route opportunities after required files exposed the imports.
  Neither reduced exploration, and both added visible-token, non-cached-input,
  and command overhead. Packet path suppression is therefore not the next
  optimization target.
- A minimal `navigation-only` follow-up retained required paths, normative law,
  and checks while omitting optional paths, route summaries, and descriptive
  implementation facts. Its 129-token packet was smaller than `full` and
  reduced average repository exposure from 6.33 to 4 paths across three paired
  Sol/xhigh runs. It still caused the model to inspect the omitted dependency in
  every Rekon run and added command and token overhead, so it remains a
  routing-only result rather than a promoted delivery policy.
- The direct-injection six-task, three-repeat local outcome corpus passes all 36
  isolated changes. All 18 pairs are candidates under the corrected evaluator;
  average explored paths fell from 25.28 to 8.28 and commands from 3.44 to
  2.28. Token usage remains uncollected.
- A pinned Sol/xhigh managed canary now captures local subscription usage. All
  six pairs preserved behavior and scope while average explored paths fell from
  34.33 to 5.17. Reported tokens fell only 1.4% and the visible estimate rose
  1.2%, so the primary result is token-neutral. Selective repeats reproduced a
  large policy-task saving and repeatable implementation/refinement overhead;
  no general token-savings claim is accepted yet.
- Instruction `1.7.3` reduced the advertised MCP schema from 3,757 to 1,730
  bytes and managed MCP steps from 2,447 to 788 bytes. The affected
  cross-system task retained full quality, eliminated unnecessary refinement,
  batched six known files into one read, and used 19% fewer reported and 24%
  fewer visible-estimate tokens than its original managed failure. A positive
  runtime-binding task still used one exact source-target resolution and
  passed. These affected-case runs do not establish general token savings.
- A fresh three-pair Sol/xhigh mixed-layout canary passed 3/3 Rekon changes
  versus 1/3 baseline and reduced average explored paths by 67%. Reported
  tokens were 26.5% higher overall. The equal-correctness pair still carried
  19.6% reported and 23.3% visible-estimate overhead, so the retained result is
  `correctness-and-exploration-gain-token-overhead`, not token savings. One
  repeat per task is not promotion evidence.
- Instruction `1.6.0` sends exact source paths to direct reads and reserves
  refinement for unresolved symbolic relationships. Finalized managed runs
  passed 12/12 complete-packet changes with full `readFirst` use and zero
  refinement calls. Average explored paths were 5.5 on the six-case corpus,
  5.67 on the mixed corpus, and 6.33 on the independent corpus.
- A symbolic implementation/consumer/producer corpus first recorded 9/9
  correct managed changes versus 3/9 controls through bounded refinement. API
  measurement then showed that waiting for the model to request a deterministic
  route doubled managed turns and increased tokens. The compiler now places at
  most two task-signaled symbolic routes in `readFirst` and reserves refinement
  for unresolved relationships. The replacement gate retains full recall and
  precision with zero refinement calls. Instruction `1.7.1` also treats an
  already-selected relationship route as resolved unless inspected source
  names another absent target; its current source-backed gate passed 3/3
  behavior and adoption checks without refinement.
- A separate runtime-binding case exercises the bounded fallback after the
  compact packet omits a known RuntimeTruth implementation. Under instruction
  `1.7.3`, the managed agent read the complete five-file initial packet,
  requested one publisher-anchored `implementation` source-target resolution,
  used the returned serializer, made only the required edits, and passed public
  and hidden checks without broad search. Its source-free calibration is
  digest-bound. A subsequent low-effort,
  provider-backed structured simulation did not reproduce that behavior: both
  pinned models made zero refinement calls across three repeats, omitted the
  serializer, and produced lower-quality plans. Luna used 7% fewer tokens than
  baseline; Sonnet used 33% more. Because that run changed both model and tool
  surface, it is scoped evidence about the simulation rather than a general
  rejection of native-agent refinement.
- In the proactive provider comparison, GPT-5.6 Luna reduced inspected files by
  34%, turns by 28%, total tokens by 28%, and estimated cost by 26% versus its
  baseline. Claude Sonnet 5 used the same files and turns as baseline with 9%
  more tokens and 7% more estimated cost. Both models used about half the
  managed tokens required by the earlier refinement round trip.
- A separate 31-file TypeScript/Python corpus records 9/9 correct Rekon changes
  versus 5/9 baseline changes across three repeats. Average explored paths fell
  from 15.89 to 6.11. Rekon preserved a cross-language producer/consumer
  contract in 3/3 runs versus 0/3 baseline runs; explicit regression-coverage
  pacts removed the only observed Rekon quality miss. Its final managed gate
  also passed 3/3 with full `readFirst` use.
- An independent 35-file layout adds configuration rollout, pathless extension
  placement, and optional cross-language schema evolution. Rekon passed 9/9
  direct changes versus 3/9 baseline, with all nine pairs candidates; average
  explored paths fell from 26.78 to 6.56 and commands from 4.11 to 2.78. Its
  final managed gate passed 3/3 with full `readFirst` use.
- The pinned GPT-5.6 Luna calibration records 18/18 full-quality Rekon plans
  versus 5/18 baseline plans. Rekon reduced average inspected files by 47%,
  turns by 29%, total tokens by 33%, and estimated standard API cost by 41%.
  Seventeen of 18 pairs were candidates; one correct cross-system repeat
  remained no-advantage because it inspected an additional compatibility
  caller.

## Next Evaluation

- Repeat repository-law evaluation only on a genuinely independent repository
  layout or task shape. The current two-task corpus has enough repeats to retain
  its bounded result but not enough breadth for cross-repository or lower-model
  claims.
- Extend positive refinement coverage only when a new relationship or task
  shape is introduced; retain separate gates for required and unnecessary
  refinement.
- Add another independent layout only when it introduces a genuinely new
  change shape, such as framework lifecycle wiring, data migration, or UI/API
  contract coordination; do not grow fixtures by duplicating service edits.
- Continue recording selected refs actually used, avoidable refs, source files
  discovered outside the packet, and whether pacts and checks were sufficient.
- Test whether the new bounded source anchors reduce avoidable whole-file reads
  or improve first-pass correctness on an independent task shape. Include one
  extension task that can use the bounded exemplar. Do not expand either into
  source bundles unless that evidence justifies the added payload.
- Expand the pinned comparison to larger mixed-layout repositories while
  retaining the same source-free, paired, repeated protocol. Treat aggregate
  efficiency as the decision signal and retain per-pair variance rather than
  tuning compiler policy to one stochastic run.
- Measure representative paired managed runs through the real Sol subscription
  interface using aggregate Codex usage and visible-payload estimates. Preserve
  correctness and first-pass scope as hard gates, then use a small high-quality
  API canary only to confirm the direction of stable local results.
- Do not invest further in role labels, lazy route materialization, or pathless
  route summaries as source-read controls. The current evidence shows that Sol
  follows both visible relative imports and hidden symbolic imports even when a
  fresh exact capability contract answers the relevant behavior question.
  Optimize task scoping, search avoidance, and delivery overhead instead of
  adding stronger skip prose. Keep source-read avoidance subordinate to
  correctness.
- Use a small high-quality API canary only after local aggregate behavior is
  stable enough to measure token direction rather than continue interface
  tuning.

## Later Expansion

- Richer language and framework packs.
- Runtime observation and drift reporting.
- Watcher-driven freshness.
- Additional CI and pull request publication surfaces.
- More mature memory curation and work-order flows.
- Permissioned source-write reconciliation for narrow, reviewable operations.
- Optional hosted or dashboard surfaces after the local substrate is solid.

Ideas in this section are direction, not a promise of shipped behavior. Current
behavior is defined by source code, CLI output, artifact schemas, and living
concept docs.
