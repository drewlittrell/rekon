# StepCapabilityGraph v1 Decision

## Decision Summary

This decision fixes the **v1 artifact shape and inputs** for
`StepCapabilityGraph`, the first artifact in the staged spine selected by
the [StepCapabilityGraph / HandoffContract Architecture
Decision](step-capability-handoff-architecture-decision.md).

**Decision: projection + optional config.** `StepCapabilityGraph` v1 is
**derived by projection** from existing governed artifacts —
`EvidenceGraph`, `CapabilityMap v2`, and `CapabilityPhraseReport` — with
an **optional** operator config at `.rekon/step-capability-map.json` used
only for grouping and labeling. Projection works with no config at all;
the config never authors the graph.

**StepCapabilityGraph v1 is an expected workflow topology graph.** It
records the *expected* step → capability → file/system structure of a
repo, with reserved placeholders for expected-handoff edges. **It does
not model runtime truth, handoff coverage, or execution readiness** —
those belong to later spine layers (`RuntimeGraphObservationReport`,
`HandoffCoverageReport`) and are explicitly out of v1.

**StepCapabilityGraph is workflow topology, not CapabilityMap v2.** The
capability map projects capabilities to subjects/systems; the graph adds
the step/workflow layer and its edges on top, without mutating the map.

**The optional `.rekon/step-capability-map.json` config is optional
grouping and labeling, not a manual-admin-heavy system.** Its absence is
the normal case; when present it only groups, labels, or merges projected
steps. **No runtime behavior changes ship in this decision.**

## Why This Decision Exists

The architecture decision reserved five artifacts and put
`StepCapabilityGraph` first because runtime drift is meaningless without
an expected graph. Before implementing it, v1 needs a fixed shape and a
clear sourcing stance so the implementation slice has an unambiguous
target — and so the artifact does not quietly drift into implying runtime
truth or becoming a hand-maintained config burden.

Two failure modes to avoid, called out explicitly:

1. **Over-claiming.** Classic grounded its step graph in runtime
   execution stats. A Rekon v1 that copied that would imply runtime truth
   / coverage / execution readiness it cannot yet provide. v1 is **static
   expected topology only**.
2. **Admin sprawl.** Classic's `step-capability-map.yaml` was an
   operator-authored source of truth. Rekon v1 must not require operators
   to hand-maintain a step map; the config is **optional grouping /
   labeling**, and projection is the default source.

## Architecture Context

From the parent architecture decision (Option B, staged spine):
`StepCapabilityGraph` (1) → `HandoffContract` (2) → `HandoffCoverageReport`
(3) → `RuntimeGraphObservationReport` (4) → `RuntimeGraphDriftReport` (5).
This decision covers **only artifact 1's v1 shape + inputs**. Handoffs,
coverage, observation, and drift remain out of scope.

## What StepCapabilityGraph v1 Is

A governed, projected artifact describing the **expected workflow
topology** of a repo: which steps/workflows realize which capabilities,
through which files and systems, and where handoffs are *expected*
(reserved, not populated in v1). It is read-only with respect to its
inputs and is regenerated by projection like other Rekon artifacts.

## Options Considered

| Option | Decision | Reason |
| --- | --- | --- |
| projection-only (no config) | rejected | no operator affordance to group/label/merge steps |
| projection + optional config | selected | projection-first; optional grouping/labeling without admin burden |
| config-only (operator authors the step map) | rejected | manual-admin-heavy; repeats classic's hand-maintained map |
| runtime-grounded v1 | rejected | implies runtime truth / execution readiness Rekon cannot yet provide |

- **Projection-only.** Rejected: gives operators no way to name, group, or
  merge projected steps for their repo's vocabulary.
- **Projection + optional config.** Selected: the graph is always
  derivable from governed artifacts; the optional config only refines
  grouping/labeling and is never required.
- **Config-only.** Rejected: reproduces classic's manual-admin-heavy
  `step-capability-map.yaml` as a source of truth.
- **Runtime-grounded v1.** Rejected: v1 would imply runtime truth /
  handoff coverage / execution readiness; runtime grounding is a later
  spine layer.

## v1 Inputs

| Input | Role | Required? |
| --- | --- | --- |
| EvidenceGraph | substrate: files, symbols, ownership / capability hints | required |
| CapabilityMap v2 | capabilities → subjects/systems (phrase-backed entries) | required |
| CapabilityPhraseReport | capability phrases → step naming / labels | optional (recommended) |
| `.rekon/step-capability-map.json` | operator grouping / labeling / merging of projected steps | optional |

Projection is driven by `CapabilityMap v2` + `EvidenceGraph`;
`CapabilityPhraseReport` improves step labels when present; the operator
config refines the result. v1 **does not mutate** `EvidenceGraph`,
`CapabilityMap`, or `CapabilityPhraseReport`.

## v1 Node / Edge Shape

| Element | Kind | v1 status |
| --- | --- | --- |
| step node | workflow/step unit (projected from capability clusters) | populated |
| capability node | capability id (from CapabilityMap v2) | populated |
| file / system node | file path / owner system (from EvidenceGraph) | populated |
| step → capability edge | `realizes` (step realizes capability) | populated |
| step → file / system edge | `touches` (step implemented in file/system) | populated |
| expected-handoff edge | declared baton placeholder | reserved (not populated in v1) |
| runtime grounding | observed execution evidence | reserved (not populated in v1) |

v1 reserves the expected-handoff-edge and runtime-grounding fields so the
later `HandoffContract` and `RuntimeGraphObservationReport` layers can
fill them, but v1 itself leaves them empty. This keeps the v1 artifact
honestly *expected/static*.

## What v1 Does Not Model

`StepCapabilityGraph` v1 does **not** model runtime truth, handoff
coverage, or execution readiness; it declares no handoff contracts,
ingests no runtime/handoff events, computes no drift, and asserts nothing
about whether a workflow can or did execute. Those are downstream spine
layers.

| Boundary | Decision |
| --- | --- |
| StepCapabilityGraph v1 vs CapabilityMap v2 | workflow topology vs capability projection |
| StepCapabilityGraph v1 vs RuntimeGraphObservationReport | expected topology vs observed runtime graph |
| StepCapabilityGraph v1 vs HandoffContract | reserved handoff placeholders vs declared handoffs |
| optional config vs source of truth | grouping / labeling vs authored graph |

## Optional Operator Config

`.rekon/step-capability-map.json` is **optional**. When absent (the
normal case), v1 projects the graph from governed artifacts alone. When
present, it may only:

- group projected steps into named workflows,
- relabel/rename steps for the repo's vocabulary,
- merge or split projected step clusters.

It is **not** a source of truth for capabilities, files, systems, or
handoffs, and it is **not** required for the artifact to be produced. It
is optional grouping and labeling, not a manual-admin-heavy system.

## Intent Impact (Deferred)

Intent integration is out of scope for this decision, but the v1 shape is
chosen so a future `intent:assess` / `intent:prepare` can attach steps
and capabilities once the artifact exists. Handoff coverage and runtime
drift inputs to intent remain deferred to later spine layers.

## What This Does Not Do

This decision implements no `StepCapabilityGraph`, registers no artifact
type, adds no CLI command, mutates no existing schema or artifact
(`EvidenceGraph` / `CapabilityMap` / `CapabilityPhraseReport`), defines no
`HandoffContract`, adds no runtime observation or drift, and starts no
intent implementation. It imports nothing from classic codebase-intel,
bumps no version, and publishes nothing. **No runtime behavior changes
ship in this decision.**

## Implementation Sequence

1. **StepCapabilityGraph v1 decision** (this memo) — shape + inputs +
   projection-first sourcing.
2. **StepCapabilityGraph v1 implementation** — register the artifact type,
   add the projection builder (inputs above), and a read-only CLI to
   produce it. Expected-handoff and runtime-grounding fields stay
   reserved/empty.
3. **HandoffContract decision + v1** — declared baton policy that fills
   the expected-handoff placeholders.
4. Later: `HandoffCoverageReport` → `RuntimeGraphObservationReport` →
   `RuntimeGraphDriftReport`, then intent spine integration.
