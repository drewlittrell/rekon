# Intent VerificationPlan Handoff

Intent VerificationPlan handoff is the proof-planning half of the Rekon intent
spine: *given a proof-approved prepared plan whose verification requirements are
proof obligations, how are those obligations turned into a runnable verification
plan?* The answer is one `VerificationPlan` artifact, generated by an explicit,
gated, non-executing handoff — `rekon intent verification-plan generate`. The
generator reads a proof-approved `PreparedIntentPlan` (gated by
`IntentStatusReport` and a handoff-time freshness / drift recheck), classifies each
requirement command for safety, and writes exactly one new `VerificationPlan`.

## Planning, Not Running

**Intent VerificationPlan handoff generates VerificationPlan only from a
proof-approved PreparedIntentPlan.** It converts approved verification
requirements into one `VerificationPlan` artifact — a list of commands and success
criteria — but it never runs those commands. The boundary statements the generator
preserves:

- **VerificationPlan generation must require PreparedIntentPlan verification
  requirements.** A plan with no `verificationRequirements` cannot produce a
  VerificationPlan.
- **IntentStatusReport gates generation but does not generate VerificationPlan.**
  Status is a read-only rollup; it authorizes the handoff by reporting a
  proof-planning-ready status, but the VerificationPlan is generated by this
  handoff, not by status reporting.
- **WorkOrder is optional in v1 and cited when available.** The handoff does not
  require a `WorkOrder`; when one is supplied it is recorded for traceability and
  status context.
- **Freshness and runtime drift are rechecked at handoff time.** A
  `PathFreshnessReport` that reports stale scoped context, or a
  `RuntimeGraphDriftReport` with new high-severity drift, blocks generation even
  if the plan was approved earlier.
- **Generated VerificationPlan traces back to PreparedIntentPlan.** Every
  generated VerificationPlan records the source `PreparedIntentPlan` (and the
  `IntentAssessmentReport`, `IntentStatusReport`, optional `WorkOrder`, freshness,
  and drift refs it was gated by) plus the requirement / phase / obligation ids.
- **VerificationPlan generation does not create WorkOrder.**
- **VerificationPlan generation does not create VerificationRun or
  VerificationResult.**
- **VerificationPlan generation does not execute commands.**
- **VerificationPlan generation does not write source files.**
- **intent:go remains deferred.** This handoff produces a plan, never a run.

## The Generation Gate

VerificationPlan generation is allowed only when **all** of the following hold.
If any fails, the handoff reports deterministic blockers and writes **no**
VerificationPlan.

| Gate | Required state |
| --- | --- |
| PreparedIntentPlan present | a proof-approved plan and its ref are supplied |
| PreparedIntentPlan approval | `approval.status === "approved"` |
| PreparedIntentPlan status | `status.value === "prepared"` |
| verification requirements | non-empty, and every command classifies safe / needs-review |
| IntentStatusReport | `status.value` is `work-ready`, `work-in-progress`, or `verification-ready` |
| blockers | no high-severity IntentStatusReport blockers |
| downstream handoff | `verificationPlanAllowed === true` and `sourceWriteAllowed === false` |
| freshness recheck | no stale scoped context at handoff time |
| drift recheck | no new high-severity runtime drift at handoff time |

## Command Safety

Verification requirement commands travel into the plan as **text only**; the
generator never executes them. Each requirement command is classified:

- **safe** — a deterministic command string (e.g. `npm run typecheck`,
  `npm test`, `node scripts/<name>.mjs`); copied into `VerificationPlan.commands`.
- **needs-review** — a commandless requirement with a clear reason, or a command
  outside the safe allowlist; surfaced as a success criterion / check, never as an
  executable command.
- **rejected** — a command containing shell-control or destructive tokens (`;`,
  `&&`, `|`, `>`, `rm`, `git push`, `npm publish`, …), or a requirement with
  neither command nor reason; **blocks generation**.

## What It Reads, What It Writes

The handoff reads a `PreparedIntentPlan` (required), an `IntentStatusReport`
(required — the latest generated when a ref is not pinned), and optionally a
`WorkOrder`, `PathFreshnessReport`, and `RuntimeGraphDriftReport`; it also reads
the plan's `IntentAssessmentReport` ref for traceability. It writes exactly one
`VerificationPlan` artifact (`source: "intent-handoff"`, with an additive
`intentHandoff` block) and nothing else: no `WorkOrder`, no `VerificationRun`, no
`VerificationResult`, and no source files. The inputs are never mutated.

## CLI

```sh
rekon intent verification-plan generate \
  --prepared-plan <PreparedIntentPlan:id|type:id> \
  [--intent-status <IntentStatusReport:id|type:id>] \
  [--work-order <WorkOrder:id|type:id>] \
  [--path-freshness <PathFreshnessReport:id|type:id>] \
  [--runtime-drift <RuntimeGraphDriftReport:id|type:id>] \
  [--root <path>] [--json]
```

When the gate passes the command writes one `VerificationPlan` and prints its ref;
when the gate fails it prints the blockers, exits non-zero, and writes no
VerificationPlan. In neither case does it create a `WorkOrder`, a
`VerificationRun`, or a `VerificationResult`, execute commands, or write source
files.
