# Intent WorkOrder Handoff

Intent WorkOrder handoff answers the boundary that follows status reporting in
the staged Rekon intent spine: *given a proof-approved prepared plan that status
reports as work-ready, how is that preparation converted into implementation
guidance?* The answer is one `WorkOrder` artifact, generated by an explicit,
gated, read-only handoff — `rekon intent work-order generate`. The generator
reads a proof-approved `PreparedIntentPlan` (gated by `IntentStatusReport` and a
handoff-time freshness / drift recheck) and writes exactly one new `WorkOrder`.

## Generating, Not Acting

**Intent WorkOrder handoff is WorkOrder artifact generation, not intent:go.** It
converts approved preparation into one `WorkOrder` artifact; it does not start,
schedule, or run the work the WorkOrder describes. The boundary statements the
generator preserves:

- **WorkOrder generation requires a proof-approved PreparedIntentPlan.** A plan
  whose approval is not `approved` cannot produce a WorkOrder.
- **IntentStatusReport gates WorkOrder generation but does not generate
  WorkOrder.** Status is a read-only rollup; it authorizes the handoff by
  reporting `work-ready`, but the WorkOrder is generated by this handoff, not by
  status reporting.
- **WorkOrder generation rechecks freshness and runtime drift at handoff time.**
  A `PathFreshnessReport` that reports stale scoped context, or a
  `RuntimeGraphDriftReport` with new high-severity drift, blocks generation even
  if the plan was approved earlier.
- **Generated WorkOrder traces back to PreparedIntentPlan.** Every generated
  WorkOrder records the source `PreparedIntentPlan` (and the `IntentStatusReport`,
  `IntentAssessmentReport`, freshness, and drift refs it was gated by).
- **WorkOrder generation does not create VerificationPlan.** Proof planning is a
  separate handoff with its own gate.
- **WorkOrder generation does not execute commands.**
- **WorkOrder generation does not write source files.**
- **intent:go remains deferred.** This handoff produces guidance, never
  execution.

## The Generation Gate

WorkOrder generation is allowed only when **all** of the following hold. If any
fails, the handoff reports deterministic blockers and writes **no** WorkOrder.

| Gate | Required state |
| --- | --- |
| PreparedIntentPlan present | a proof-approved plan and its ref are supplied |
| PreparedIntentPlan approval | `approval.status === "approved"` |
| PreparedIntentPlan status | `status.value === "prepared"` |
| PreparedIntentPlan next action | `status.recommendedNextAction === "create-work-order"` |
| IntentStatusReport | `status.value === "work-ready"` |
| blockers | no high-severity IntentStatusReport blockers |
| downstream handoff | `workOrderAllowed === true` and `sourceWriteAllowed === false` |
| phases | the prepared plan has at least one phase |
| freshness recheck | no stale scoped context at handoff time |
| drift recheck | no new high-severity runtime drift at handoff time |

## What It Reads, What It Writes

The handoff reads a `PreparedIntentPlan` (required), an `IntentStatusReport`
(required — the latest generated when a ref is not pinned), and optionally a
`PathFreshnessReport` and `RuntimeGraphDriftReport` for the handoff-time recheck;
it also reads the plan's `IntentAssessmentReport` ref for traceability. It writes
exactly one `WorkOrder` artifact and nothing else: no `VerificationPlan`, no
`VerificationRun`, no `VerificationResult`, and no source files. The inputs are
never mutated.

The generated WorkOrder maps the prepared plan into implementation guidance:
the request goal becomes the WorkOrder goal; phase paths and systems become the
WorkOrder paths and owner systems; obligations and blocked reasons become risk
notes; verification requirements become required checks; and a fixed boundary
section records that the WorkOrder was generated by an Intent WorkOrder handoff
that creates no VerificationPlan, executes no commands, and writes no source
files. The WorkOrder's `source` is `intent-handoff`, and its `intentHandoff`
block carries the source refs and the phase / obligation / verification ids.

## Blockers

When the gate fails, the handoff returns a `blocked` result whose blockers each
carry a stable `category`: `missing-prepared-plan`, `plan-not-approved`,
`plan-not-prepared`, `next-action-not-work-order`, `status-not-work-ready`,
`status-has-high-blocker`, `handoff-not-allowed`, `source-write-boundary`,
`empty-phases`, `freshness-stale`, `drift-changed`, and `missing-source-ref`. A
blocked handoff sets a non-zero exit code and writes no WorkOrder.

## CLI

```sh
rekon intent work-order generate \
  --prepared-plan <PreparedIntentPlan:id|type:id> \
  [--intent-status <IntentStatusReport:id|type:id>] \
  [--path-freshness <PathFreshnessReport:id|type:id>] \
  [--runtime-drift <RuntimeGraphDriftReport:id|type:id>] \
  [--root <path>] [--json]
```

When the gate passes the command writes one `WorkOrder` and prints its ref; when
the gate fails it prints the blockers, exits non-zero, and writes no WorkOrder.
In neither case does it create a `VerificationPlan`, execute commands, or write
source files.
